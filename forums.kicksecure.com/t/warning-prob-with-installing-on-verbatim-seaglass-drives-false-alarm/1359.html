Generally, USB drives themselves act like any other drive under Linux; it‚Äôs theoretically possible a drive could be compatible with Windows and not Linux, but in practice I have never seen this be the case.

You‚Äôre using Rufus to burn the ISO to the drives. Rufus might not install Kicksecure to a USB drive properly; the ISO is designed to be written directly to a disk, not unpacked into a filesystem on the disk. If you must use Rufus, try using ‚Äúdd mode‚Äù to write the ISO. Better yet, use an existing Linux installation and flash the ISO using dd, i.e. sudo dd if=./kicksecure.iso of=/dev/sdX bs=4M.

(There is, of course, always the possibility that the drives themselves are failing. Drives are most likely to fail when they‚Äôre brand new and when they‚Äôre relatively old. There are command-line utilities for Linux such as badblocks you can use to check this, see man badblocks.)

CodeLurker:

In each failure this time, and most of the previous ones, it gave the error about bootloader_config not finishing after 600 secs.

That sounds like the drives you‚Äôre using are so slow that the default timeout value isn‚Äôt long enough; I‚Äôve seen these errors when installing to some USB 3 TeamGroup drives in the past. Many USB flash drives have decent read speed but horrible write speed, even USB 3 drives.

The good solution would be to use faster drives, but if you need to use the drives you have, you can lengthen the timeout:

  • Boot the live ISO, and select LIVE Mode | SYSMAINT Session | system maintenance, install.
  • Click Open Terminal.
  • Run sudo nano /usr/lib/calamares/modules/bootloader-config/module.desc.
  • Adjust the timeout: 600 line to something longer, like timeout: 5000.
  • Save with Ctrl+S, exit with Ctrl+X.
  • Close the terminal window.
  • Click Install System.
  • Install like normal.

Because USB installations like you‚Äôre doing are explicitly supported, and USB drives are so frequently slow, we should probably lengthen some of these timeouts.
