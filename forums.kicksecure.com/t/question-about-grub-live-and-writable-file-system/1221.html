arraybolt3 August 13, 2025, 10:33pm 2

The current live mode detection code isn’t Veracrypt-volume aware, so it can’t tell the difference between a container located on an ephemeral live filesystem, and a container located on a writable disk device. It just sees that you have a writable mounted device and warns you about it.

It might be possible to add code that is aware of Veracrypt volumes, but that would only make it work with Veracrypt specifically. There are a lot of applications that may mount a writable container that our code would have to be aware of, so in general this isn’t an easy problem to solve.

If the container itself is located on an ephemeral filesystem, then your writes to it should disappear on reboot, just like with any file on the ephemeral filesystem.

arraybolt3 August 17, 2025, 4:00pm 9

Reproduced issue on Kicksecure 17.4.4.6 using VirtualBox with EFI enabled.

Looks like the issue is that live-hardener tries to remount /boot/efi read-only after it mounts a read-write overlay on /boot. Mounts don’t “show through” an overlayfs, so /boot/efi isn’t even accessible in live mode, and remounting /boot/efi doesn’t do anything in this situation.

Created a fix at GitHub - ArrayBolt3/grub-live at arraybolt3/live-hardener-fix which is pending review. I’m unsure if this will end up in Kicksecure 17 or if it will have to wait until Kicksecure 18, but there’s a known fix now.

arraybolt3 August 19, 2025, 2:34am 11

Could you share a screenshot of the livecheck window so I can see the message you’re getting?

Yes, I am sure it works either the same or better than previously. The previous live mode implementation simply used a Dracut module to mount the root filesystem read-only, then put an in-memory overlay on it. That meant that any filesystem mounted thereafter (like /boot, /boot/efi, /home if you installed it on a separate partition, etc.) would still be read-write. The thing that changed is that livecheck now detects these other writable filesystems and reports about them, and a new service (live-hardener) attempts to remount more filesystems read-only and then mounts in-memory overlays on them. The original, dracut-based root overlay is still in place, so worst-case-scenario, live mode works as good or better than it used to.
