This will be used by Whonix 18, since our default file manager will be PCManFM-Q…t. Some rough spots that may need ironed out: * This reuses the `qvm-actions.sh` script from qubes-core-agent-thunar. At the moment this is done by simply copying `qubes-rpc/thunar/qvm-actions.sh` into the `qubes-core-agent-pcmanfm-qt` package, which is horrible. Duplication isn't a workable solution though because the package build procedure installs all of the files from all of the agent packages at once and then splits them into binary packages, and both `qubes-core-agent-thunar` and `qubes-core-agent-pcmanfm-qt` place `qvm-actions.sh` at the same location on the filesystem. This shared location also means that `qubes-core-agent-thunar` and `qubes-core-agent-pcmanfm-qt` have to conflict with each other. Ideally this script would be located in some shared package, but I'm not sure there's an appropriate one already existing, and I'm not sure making a new package shared only between `qubes-core-agent-thunar` and `qubes-core-agent-pcmanfm-qt` is warranted when this is just one script file. * The .desktop files used to define the actions that can be performed on each file have to use mime types to specify what files those actions are intended to match. As it turns out, there is no good way to express "match all files regardless of type, but do not match directories" as a mime type or combination thereof, so instead we currently just use "match everything" globs even on actions that don't work on directories, then display a popup warning if a directory is detected. I do not know of any good alternative to this. * Because file types are specified by mime type rather than file name extension, there is likely some introspection done on the files to determine what actions to display. I have no idea how secure Qt's mime type detection routines are, or if pcmanfm-qt even uses those routines for mime type detection as opposed to using something like `file`. I also don't know if there's a way to turn that detection logic off, and even if there was, that would break the ability to do things like only showing "Convert PDF in disposable qube" for PDF files.

`XDG_DATA_DIRS` is reset to a minimal list of directories in applications opened from the Qubes app menu, breaking default application resolution

opened 03:43PM - 05 Oct 25 UTC
ArrayBolt3
P: default

[How to file a helpful issue](https://www.qubes-os.org/doc/issue-tracking/) ###… Qubes OS release R4.3 ### Brief summary Whonix modifies the `XDG_DATA_DIRS` environment variable via scripts in `/etc/profile.d` (and `/etc/zprofile.d`). These modifications allow Whonix to customize things such as default application associations. These changes are picked up by Qubes OS and do appear in login shells, but somehow `XDG_DATA_DIRS` (and to my awareness, *only* `XDG_DATA_DIRS`) is reset to a "minimal" set of directories in applications that are opened from the Qubes app menu. This results in all of Whonix's file association settings usually being ignored. ### Steps to reproduce * Install a Whonix-Workstation 18 template. * Create an AppVM based off this template, and launch QTerminal in it. * Run `echo $XDG_DATA_DIRS`. * Open Qube Manager, select the Whonix-Workstation 18 AppVM, and click "Open console in qube". * Log in as "user" (no password required), then run `echo $XDG_DATA_DIRS`. ### Expected behavior In both instances, `echo $XDG_DATA_DIRS` should output `/usr/share/usability-misc/xdg-override/:/usr/share/anon-apps-config/share/:/usr/share/open-link-confirmation/xdg-override:/var/lib/flatpak/exports/share:/usr/local/share/:/usr/share/` or similar. ### Actual behavior In the DispVM console, `echo $XDG_DATA_DIRS` outputs the expected value, but in QTerminal, the returned string is `/home/user/.local/share/flatpak/exports/share:/var/lib/flatpak/exports/share:/usr/local/share/:/usr/share/`. ### Additional information This is contributing to an issue, where clicking on the Qubes Domains widget in the system tray, hovering over a Whonix-Workstation 18 qube, and clicking "Open File Manager" opens Catfish File Search rather than PCManFM-Qt. Whonix ships a file at `/usr/share/usability-misc/xdg-override/applications/defaults.list` that is supposed to set the default application for `inode/directory` to `pcmanfm-qt.desktop`. Right now that file is out-of-date and is pointing to `thunar.desktop` instead, but even when it is changed to point to `pcmanfm-qt.desktop`, Catfish still opens when clicking this button in the widget. Running `xdg-open $HOME` in QTerminal also opens Catfish. However, running `DISPLAY=:0 xdg-open $HOME` in the DispVM console opens PCManFM-Qt as expected. (Granted, it looks awful because `QT_QPA_PLATFORMTHEME='lxqt'` is strangely not being set properly, but that's an unrelated issue.) The only thing I see in Qubes' code that looks like it modifies `XDG_DATA_DIRS` is https://github.com/QubesOS/qubes-core-agent-linux/blob/bffc6e0d0df3874781c0feda8297be5063fabba1/qubes-rpc/qubes.StartApp. Could this be part of the issue?
